# Render Hosting Guide

This repo is ready for a single Render Docker web service that serves both the API and the static frontend.

## 1. Push the Repo

Push the current branch to GitHub first:

```sh
git push origin main
```

Render can auto-deploy from a connected GitHub branch.

## 2. Create the Render Service

Preferred path:

1. Open Render Dashboard.
2. Create a new Blueprint from this GitHub repo.
3. Select `render.yaml` at the repo root.
4. Render will create the `pulse-backend` Docker web service.

The service uses:

- Dockerfile: `backend/Dockerfile`
- Docker context: repo root
- Health check: `/api/v1/health`
- Persistent disk: `/app/backend/data`

## 3. Set Environment Variables

Set these in the Render service environment. Do not commit real values to Git.

Required:

```txt
JWT_SECRET=<generated by Render if using blueprint>
GOOGLE_CLIENT_ID=<your Google OAuth web client id>
CORS_ORIGINS=https://your-render-service.onrender.com
```

AI providers. Set at least one:

```txt
GROQ_API_KEY=
GROQ_MODEL=llama-3.3-70b-versatile
OPENROUTER_API_KEY=
DEEPSEEK_API_KEY=
HUGGINGFACE_API_KEY=
OPENAI_API_KEY=
```

Optional market/news integrations currently exposed in platform status:

```txt
NEWS_API_KEY=
FINNHUB_API_KEY=
FINNHUB_WEBHOOK_SECRET=
ALPHA_VANTAGE_API_KEY=
FMP_API_KEY=
EOD_API_KEY=
FRED_API_KEY=
MARKETAUX_API_KEY=
```

Production recommendation:

```txt
ALLOW_GUEST=0
NODE_ENV=production
```

## 4. Verify Deployment

After deploy, open:

```txt
https://your-render-service.onrender.com/api/v1/health
https://your-render-service.onrender.com/api/v1/status
https://your-render-service.onrender.com
```

Expected:

- `/api/v1/health` returns `{ "ok": true }`.
- `/api/v1/status` shows which AI providers and integrations are configured.
- The root page loads the Pulse app.

## 5. Google OAuth Setup

In Google Cloud Console, add the Render URL to the OAuth web client:

```txt
Authorized JavaScript origins:
https://your-render-service.onrender.com
```

Then set the same client ID as `GOOGLE_CLIENT_ID` on Render.

## 6. Notes

- The SQLite database lives on the Render persistent disk. This is fine for an MVP, but move to Postgres before scaling to multiple instances.
- Uploaded avatars, AI images, and games currently live on local disk. Move these to S3/R2/GCS before serious production use.
- YouTube features depend on `yt-dlp` and `ffmpeg`, both installed by the Dockerfile.
- The web proxy is stronger now, but some modern websites will still block rewrite-based browsing because of strict origin checks, DRM, service workers, or anti-bot systems.
